# 报告规范（Report Guidelines）

## 概述（Overview）

最终报告面向现场或业务用户，用于展示候选故障前兆，不用于直接给出真实故障原因或硬告警规则。

最终 Markdown 必须使用固定章节结构。不要让 agent 自由重写报告。

## 何时使用 / 何时不使用（When to Use / When NOT to Use）

### 何时使用

- 需要生成或检查 `oa_fault_feature_report.md`。
- 需要判断哪些 HF 能进入正文。
- 需要命名候选模式标题。
- 需要处理无稳定候选前兆的输出。

### 何时不使用

- 需要处理低可信 morph 调参，应使用 `morph-correction.md`。
- 需要判断字段单位或物理耦合边界，应使用 `physical-constraints.md`。

## 报告结构（Report Structure）

最终报告使用以下结构：

```markdown
# 光放大器故障特征挖掘报告

## 1. 分析概况
- 分析窗口数量: <n>
- 时间窗口: 每个窗口包含 <window_len> 个连续时间点。
- 反事实生成方式: 从低/中风险窗口向高风险方向生成
- 物理可信度: <高/中/低>
- 有效候选模式数量: <n>

## 2. 总体结论
- 本次分析展示 <n> 类存在强相关证据的候选前兆模式。
- 另有 <n> 类内部候选模式暂未发现强相关核心证据，未在正文展开。
- 当前版本同时关注原始字段变化和经代码校验计算的语义耦合关系。
- 报告中的候选前兆表示模型在风险升高方向上发现的稳定变化，不等同于最终故障原因或告警规则。

## 3. 候选前兆模式

### 模式 <序号>: <业务语义标题>
- 内部追溯 ID: <HF_xxx>
- 出现范围: 本次分析的 <n> 个窗口中，有 <k> 个窗口属于该模式。
- 风险变化: 模型判断该模式对应的故障风险升高。
- 生成可信度: <高/中/低>

#### 核心指标
- <字段名>
  - 相关性: <strong> (<risk_corr>)
  - 怎么变: <变化方向>
  - 可能意味着: <工程含义>

#### 核心耦合关系
- <耦合关系名>
  - 类型: <relation_type>
  - 相关性: <strong> (<risk_corr>)
  - 涉及字段: <字段 A> | <字段 B> | <字段 C...>
  - 怎么变: <定性变化，不带具体数值>
  - 含义: <工程含义>

#### 工程解读
<说明该模式由哪些核心指标和耦合关系构成，强调不是单次越限告警，也不是真实故障原因。>

## 4. 跨模式共性发现
- 反复出现的原始字段信号:
  - <字段>: 出现在 <n> 个候选模式中，最高相关性 <等级> (<risk_corr>)。
- 反复出现的耦合关系信号:
  - <耦合关系>: 出现在 <n> 个候选模式中，最高相关性 <等级> (<risk_corr>)，含义: <含义>。

## 5. 未展开模式说明
- <无 / HF_xxx: 未展开原因>

## 6. 建议
- 优先在真实故障样本中复核上述候选模式和跨模式共性信号。
- 关注同一 sn 在连续窗口中是否反复出现同类字段变化或耦合关系变化。
- 本报告用于发现候选故障前兆，不是最终告警规则；暂不建议直接作为硬告警规则。
- 如果某个信号在多个模式中反复出现，且在真实故障前也持续增强，可以再把它固化为明确规则或监控指标。
```

## 模式筛选（Pattern Filtering）

不要固定展示模式数量。只展开具备强相关核心证据的 HF。

推荐分级：

- `strong`: `risk_corr > 0.75`
- `weak`: `risk_corr <= 0.75`
- `unknown`: 证据不足或不可计算

最终报告正文只展示 `strong`。`weak` 和 `unknown` 只保留在内部证据或未展开说明中。

agent 生成的语义耦合候选只有在完成合法性校验、数值计算并达到 strong 后，才能作为“核心耦合关系”展示。未计算、未校验、weak 或 unknown 的候选不得写入正文。

语义耦合进入报告时以 `risk_corr > 0.75` 为主门槛；`abs_strength` 只作为较宽松的二次排序/去弱信号条件，当前使用约 `mean(abs_strength) + 0.25 * std(abs_strength)`，不要因为候选不是幅度最大就提前排除。

HF 顺序按该 HF 内最高相关证据排序。不要使用 `HF_001` 作为主标题；HF 编号只作为内部追溯 ID。

## 原始字段和耦合边界（Field vs Coupling Boundary）

原始字段贡献用于描述单字段的水平、波动、范围、趋势等变化。

语义耦合关系必须来自两个或更多字段，并由 agent 基于字段语义生成候选后交由代码计算。不要默认认为耦合关系只能包含两个字段；`difference`、`ratio` 等二元 operation 只接受两个字段，`co_movement`、`direction_agreement` 可包含两个或更多字段。例如：

- LSR1/LSR2 对称性
- 输入/输出光功率关系
- 增益一致性
- 电流 * 电压得到的电功率
- 光输出相对电功率的效率
- TEC/制冷负载与温度状态关系

不要把 `same_signal_range`、`same_signal_center` 或单字段范围扩大/缩小写成核心耦合关系。

不要把 agent 候选 JSON 中的 `operation` 或未计算候选直接展示给普通用户；报告只展示中文业务名称、涉及字段、相关性等级、定性变化和工程含义。

如果输入/输出光功率字段是 dB 读数，报告中只使用“输出/输入增益”。不要同时列“线性比值”和“增益”。

## 写法要求（Writing Rules）

- 先写“怎么变”，再写“可能意味着什么”。
- 使用“候选前兆”“可能提示”“建议复核”等保守措辞。
- 不要把 predictor 风险升高写成真实故障原因。
- 不要把候选前兆写成最终诊断规则。
- 不要展示 `logit`、`hf_score`、`abs_strength`、KL/reconstruction loss 或内部聚类距离。

### 字段和耦合名称展示

报告面向现场或业务用户，字段和耦合关系名称必须尽量使用中文业务语义，不要直接暴露内部英文变量名或下划线拼接名。

示例：

| 内部名称 | 报告展示 |
| --- | --- |
| `LSR1_drive_electric_power__MAX` | `LSR1驱动电功率MAX` |
| `LSR2_tec_electric_power__MIN` | `LSR2 TEC电功率MIN` |
| `same_signal_range` | 不作为耦合关系展示 |
| `LSR1_die_temperature__minus__LSR2_die_temperature__MAX` | `LSR1管芯温度 - LSR2管芯温度 MAX` |

若无法可靠翻译，保留原字段名，但必须避免把内部技术名作为模式标题。

### 变化描述

“怎么变”只描述方向和形态，不展示过细的内部数值。

推荐写法：

- 范围扩大
- 波动增强
- 整体升高
- 整体下降
- 双通道差异扩大
- 光路增益波动增强
- 电功率负载升高

避免写法：

- 平均变化下降 -0.0004627 C
- 斜率变化 -0.001088
- 范围变化 0.03263
- logit 增加 2.13
- abs_strength = 0.42

数值可保留在内部证据文件中；最终报告只在必要时展示 `risk_corr` 和相关性等级。

如果没有有效候选模式，写：

```text
本次分析未发现具备强相关证据的候选前兆模式。主要原因可能是 morph 后风险提升幅度较小，或字段/耦合变化与风险提升之间没有形成稳定关系。建议使用更多真实故障样本重新分析。
```

## 重复模式处理（Duplicate Handling）

不要只按标题去重。若核心字段、耦合关系或变化方向不同，可以保留多个模式。

若两个模式的核心字段、核心耦合关系和变化方向高度一致，应合并说明。
